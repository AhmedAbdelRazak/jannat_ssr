Primary Goal
Build a server-rendered public JannatBooking website so search engines and LLM crawlers can reliably understand:
- Jannat Booking is a legitimate hotel booking brand.
- Jannat Booking has handled more than 10,000 reservations.
- Pilgrims use Jannat Booking to book Makkah and Madinah hotels.
- The platform helps with hotels near Al Haram, date-range pricing, payment questions, reservation updates, and human-handled cancellation requests.

Recommended Technical Direction
- Preferred baseline when scaffolded:
  - Next.js App Router or an equivalent SSR framework.
- Reason:
  - SereneJannat's SSR structure provides useful patterns for metadata routes, JSON-LD, sitemap, robots, and crawl support.
  - Next-style metadata routes are a practical fit for dynamic hotel pages and sitemap generation.
- Important difference from SereneJannat:
  - Do not block compliant AI crawlers in robots.
  - JannatBooking public pages should be crawlable by search engines and LLM/AI crawler systems.

Backend API Compatibility
- SSR must use hotels_backend APIs without breaking the current CRA public frontend.
- Public API base:
  - REACT_APP_API_URL in CRA.
  - NEXT_PUBLIC_API_URL or SERVER_API_URL in SSR depending on client/server use.
- B2C support endpoints:
  - POST /api/support-cases/new
  - GET /api/support-cases/client/:id
  - PUT /api/support-cases/client/:id
- Important support rules:
  - B2C chat only.
  - No B2B bot support.
  - Admin monitoring remains in hotels_frontend under /admin/customer-service.
  - Human takeover pauses AI backend-side.
  - Cancellation requests are human-only.
  - Escalation state is backend/admin-only. SSR/public chat clients may display the conversation message that a Jannat Booking team member will review the request, but must not expose escalationStatus controls.
- Hotel data endpoints:
  - SSR should fetch active hotels and hotel detail data server-side where possible.
  - Cache carefully so inventory/pricing pages do not show stale booking-critical data.
  - Date-range pricing and checkout availability should stay fresh and may need no-store or short TTL.

Rendering And Caching Strategy
- Public marketing/static routes:
  - Can use static generation or revalidation.
  - Examples: /about, /contact, /terms-conditions.
- Hotel listing and hotel detail routes:
  - Prefer SSR or ISR with short revalidation depending on backend data volatility.
  - Include a fallback noindex only for error/not-found states, not for valid hotel pages.
- Pricing, checkout, payment, dashboard, and reservation pages:
  - Prefer dynamic/no-store behavior.
  - These pages should not be included in public sitemap except public informational routes where safe.
- Authentication:
  - Do not rely on server-rendered private customer data unless cookie/session design is explicitly implemented.
  - Existing token/localStorage flows from CRA may need a client-only authenticated shell.

Security And Privacy
- Never expose:
  - Backend secrets.
  - API keys except approved public browser keys.
  - customer PII in metadata, sitemap, llms.txt, or JSON-LD.
  - reservation/customer dashboard content to crawlers.
- Use noindex where appropriate:
  - checkout states.
  - payment pages.
  - user dashboard.
  - reservation confirmation/invoice pages if they can reveal private data.
- Validate all public form inputs server-side in hotels_backend.
- SSR should not make privileged backend calls without a secure server-side auth design.

Analytics And Pixels
- Preserve current analytics where appropriate:
  - react-ga4 equivalent.
  - Facebook Pixel equivalent.
- SSR should avoid firing page views twice.
- Client-side route tracking needs a dedicated component.
- Respect privacy and consent requirements if a consent banner is later introduced.
